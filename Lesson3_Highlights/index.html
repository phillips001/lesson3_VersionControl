Highlight #2 Snapshots, Not Differences

Git refers to this feature as one of the most important aspects that differentiate themselves from other Version Control Systems. Instead of thinking of version tracking of a set of accumulated differences from the original, Git tracks versions as a stream of snapshots over time.

Git refers to its tracked objects as ‘blobs’, which is a known acronym for binary large object. A blob could contain most anything: videos, sound tracks, data files, or coded programs. All of these items can be part of an overall application. And, they can and will have changes made over time. Tracking them as a snapshot with appropriate identification and timestamping of the ‘blob’ itself, rather than attempting to track the individual differences within the blob, has to be a better way.

I must admit, though, I don’t fully appreciate this feature because I haven’t had the experience working with Git. Hopefully, that understanding will develop. As an application developer for a good part of my career, versioning was never a big topic. It was important, however, and the ability to manage and track versions of software was always a requirement. Spending a lot of time on it, however, was not.

On the other hand, I’ve never been in a situation where development was a community effort. I can see that a project of this size would require a version control system that has all the bells-and-whistles available.

Highlight #3 Staging Areas

The stage is the place where it all comes together. Actually, staging is used quite a bit in IT. A company transferring data back-and-forth may use ftp as a staging area for files. A router can be thought of as a staging area for data to-and-from your pc. Git uses a staging area as an integral part of it version control – it is that in-between place where everything comes together.

You may checkout the project to your workspace. Changes and rudimentary testing are only done in your workspace. Completed changes go into the git staging area. They don’t become part of the repository until they are committed.

A staging area is important because other processes can work on it. They may deal with security (data arriving or leaving the company), speed (transferring multiple files instead of just one), or efficient workflow (integrity checks of data before it moves somewhere else). Git uses this staging area to properly identify and index changes of your completed work prior to commit.

Highlight #4 Distributed Workflow

Distributing data to localized workspaces is an inherent design attribute of git. To keep things fast and not require a dedicated server connection, project data is distributed. Actually, when git refers to the repository, it is talking about a local repository. You can commit all day to the local repository, even when you have no connection to the server. One can really appreciate this if they’ve experienced their cloud-based centralized system down for whatever reason (hurricane Irma comes to mind).

Highlight #5 A Stupid Content Tracker

So, I looked up git in Wikipedia and found version 1.0 came out in January, 2006. It was written by Linus Torvalds who also responsible for Linux. The readme file of the original version’s source code, and evidently, still retrievable, quotes:

    The name "git" was given by Linus Torvalds when he wrote the very first version. He described the tool as "the stupid content tracker" and the name as (depending on your way):

    • random three-letter combination that is pronounceable, and not actually used by any common UNIX command. The fact that it is a mispronunciation of "get" may or may not be relevant.
    • stupid. contemptible and despicable. simple. Take your pick from the dictionary of slang.
    • "global information tracker": you're in a good mood, and it actually works for you. Angels sing, and a light suddenly fills the room.

When he started writing this version control system, however, he made it a point that git was not going to act like other systems on the market, and that it needed to be capable of handling large-scale software collaboration projects, like Linux itself.
